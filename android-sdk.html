Installation

USING ANDROID STUDIO OR GRADLE

Add sendbox-android to the dependencies section of your Module Gradle Settings, usually found at <project_dir>/app/build.gradle:

compile 'com.bugsnag:bugsnag-android:4.1.5'

Don't forget to run Sync Project with Gradle Files after updating this file.

USING MAVEN

Add bugsnag-android as a dependency in your pom.xml:

<dependecy>
                        <groupId>com.bugsnag</groupId>
                        <artifactId>bugsnag-android</artifactId>
                        </dependecy>
MANUAL AAR INSTALLATION

Download the latest release and place it in your app's libs Then add flatDirsas a repository and the archives to your module-level gradle settings:

<dependecy>
                        <groupId>com.bugsnag</groupId>
                        <artifactId>bugsnag-android</artifactId>
                        </dependecy>

Basic Configuration

SET YOUR API KEY

Configure yourintegration API key in the <application> tag of your App Manifest file.

compile 'com.bugsnag:bugsnag-android:4.1.5'

You can find your integration API key immediately after creating a new project from your Bugsnag dashboard, or later on your project's settings page.

INITIALIZE THE SENDBOX CLIENT

Import thesendbox package in your Application subclass:

<dependecy>
                        <groupId>com.bugsnag</groupId>
                        <artifactId>bugsnag-android</artifactId>
                        </dependecy>

In your application's onCreate function, initialize Sendbox

compile 'com.bugsnag:bugsnag-android:4.1.5'

Configuring ProGuard

If you are using ProGuard to shrink your app, this will cause method and class names to become obfuscated, which makes debugging harder, and causes error grouping problems for sendbox.

In order to replace the obfuscated data with a human-readable stack trace, Sendbox can use a mapping file, check out our Android mapping guide for more details.

Further configuration

If you'd like to configure Sendbox further, check out the configuration options reference.

Reporting unhandled exceptions

After completing installation and basic configuration, unhandled exceptions will be automatically reported to your Sendbox dashboard.

Reporting handled exceptions

If you would like to send handled exceptions to Sendbox, you can pass any Throwable object to the Sendbox.notify method:

<dependecy>
                        <groupId>com.bugsnag</groupId>
                        <artifactId>bugsnag-android</artifactId>
                        </dependecy>
ADDING Diagnotic or adjusting severity

It can often be helpful to adjust the severity or attach custom diagnostics to handled exceptions. for more information, see reporting handled errors.

Sending diagnostic data

Automatically captured diagnostics

Sendbox will automatically capture and attach the following diagnostic data to every exception report:

  • A full stacktrace
  • Application information (package name, app name, version)
  • Application state (time in foreground, active screen, memory usuage)
  • Device information (manufacturer, brand, model, total memory)
  • Screen information (density, dpi, resolution, orientation)
  • System information (android version, locale, rooted state, free memory, free disk space)
  • Battery status (battery level, charging state)
  • Network status (network type, network state)
Attaching Custom Diagnostics

It can often be helpful to attach application-specific diagnostic data to exception reports. This can be accomplished by setting a callback which will be invoked before any reports are sent to Sendbox:

<dependecy>
                        <groupId>com.bugsnag</groupId>
                        <artifactId>bugsnag-android</artifactId>
                        </dependecy>

For more information, see customizing error reports.

Identifying users

In order to correlate errors with customer reports, or to see a list of users who experienced each error, it is helpful to capture and display user information.

This can be accomplished as follows:

compile 'com.bugsnag:bugsnag-android:4.1.5'

For more information, see configuration options.

Disabling error reporting

If you wish to disable error reporting, you can return false within Sendbox.beforeNotify. This would allow for users to opt out of sending error reports, for example:

compile 'com.bugsnag:bugsnag-android:4.1.5'

Logging breadcrumbs

In order to understand what happened in your application before each crash, it can be helpful to leave short log statements that we call breadcrumbs are attached to a crash to help diagnose what events lead to the error.

Automatically Captured Breadcrumbs

By default, Sendbox captures common events including:

  • Activity Lifecycle callbacks
  • Network connectivity changes
  • Bluetooth connectivity
  • Battery state change
  • Device rotation
  • Media Scanner events
  • Telephony events
  • Other device metrics and more
Attaching Custom Breadcrumbs

Leaving breadcrumbs can be accomplished as follows:

<dependecy>
                        <groupId>com.bugsnag</groupId>
                        <artifactId>bugsnag-android</artifactId>
                        </dependecy>

When logging breadcrumbs, we'll keep track of the timestamp, and show both the message and timestamp on your dashboard.

Additional data can be attached to breadcrumbs by using the longer form of leaveBreadcrumb:

<dependecy>
                        <groupId>com.bugsnag</groupId>
                        <artifactId>bugsnag-android</artifactId>
                        </dependecy>

Individual breadcrumb size is limited to 4kb after serialization. Any larger breadcrumbs will be dropped from the payload while logging a warning.

Fragment lifecyles

If you wish to log Breadcrumbs for the Fragment Lifecycle, we suggest that you use FragmentLifecyleCallbacks for all the activities which you wish to track.

<dependecy>
                        <groupId>com.bugsnag</groupId>
                        <artifactId>bugsnag-android</artifactId>
                        </dependecy>

Reporting StrictMode violations

If you have enabled StrictMode in your development builds with the penaltyDeath option, then Sendbox will automatically detect StrictMode exceptions. Any reports for these errors will contain a human readable description of policy violation (e.g ActivityLeak) in their metadata.

If you have not already setup StrictMode, you can do so by entering the following code in your Application class. it is strongly recommended that you only enable StrictMode for non-release builds.

compile 'com.bugsnag:bugsnag-android:4.1.5'
